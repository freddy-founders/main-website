1. Events

Default public front page: compact HN-like list of upcoming events with RSVP/register.

PublicDefaultHN-like

User Stories

  • As a visitor, I can immediately see what events are coming up.
  • As a founder/member, I can scan upcoming events and decide what is relevant to me.
  • As a visitor, I can RSVP/register without digging through email.
  • As a returning member, I can distinguish upcoming, full, closed, and past events quickly.
  • As an organizer, I can share one URL as the canonical events source.

Affordances

  • Compact text rows.
  • Date/month separators.
  • Status tags: open, full, closed, past.
  • RSVP/Register button or link.
  • Details link only when needed.
  • Optional people signal: “host + 2 relevant people”.
  • Clear archived/past treatment.

Row Content

  • Date/time.
  • Plain event title.
  • Location or format.
  • Host.
  • One-line fit/context.
  • Status.
  • RSVP/Register action.

Page Design

  • Top bar: Freddy Founders, Events, People, Companies, optional Login.
  • No title banner: page title, subtitle, and lightweight stats should move into the right sidebar on desktop.
  • Primary list: dense upcoming-first rows.
  • Row hierarchy: title first, metadata second, RSVP visible but not flashy.
  • Metadata uses small muted/mono text.
  • Right sidebar: spec-aligned rail for context, filters, visible fields, stats, and boundaries.

Right Sidebar

  • Page context: what Freddy Founders Events is.
  • Stats: upcoming events, public people, event-linked companies if useful.
  • Filters: upcoming, open, in person, remote, past.
  • Visible fields: date/time, location/format, host, status, fit/context, RSVP/Register, details when useful.
  • Boundaries: no feed/comments/voting, no payment/ticketing flow, RSVP only if likely attending.

Do Not Include

  • No hero bloat.
  • No feed/posts/comments.
  • No voting/ranking/karma.
  • No payment/ticketing flow.
  • No marketing landing-page treatment.

2. Event Detail

Optional page for events that need “why attend?” and “who will I meet?” context.

OptionalContextRSVP

User Stories

  • As a visitor, I can understand whether this event is for me before I RSVP.
  • As a founder/member, I can see who is relevant to meet at this event.
  • As a visitor, I can understand the event format and expectations.
  • As an organizer, I can attach public-safe people context to an event.
  • As an organizer, I can keep event context concise and current.

Affordances

  • Prominent RSVP/Register action.
  • Back to Events link.
  • Shareable event URL.
  • Status tag.
  • People-at-event compact rows.
  • Relevance labels: host, speaker, attending, good to ask about X.

Content

  • Title, date/time, location/format, host.
  • One-line fit/context.
  • Short “why attend?” copy.
  • Intended audience / what to expect.
  • Event status.
  • RSVP/register action.
  • People-at-event module when useful.

Page Design

  • No title banner: event name, short explanation, and lightweight stats should sit in the right sidebar on desktop.
  • Primary action zone near the top of the main content; repeat near bottom only if long.
  • Context block: short, practical, non-marketing.
  • Fit block: who should attend / expectations.
  • People block: compact rows with relevance labels.
  • Desktop uses the right sidebar for event overview and stats; mobile is one column.

Do Not Include

  • No discussion thread.
  • No attendee social feed.
  • No full event platform.
  • No long marketing page.
  • No people context unless tied to this event.

3. People

Bookface-like public-safe directory/index of people in and around Freddy Founders.

DirectoryBookfacePublic-safe

User Stories

  • As a visitor, I can see who is in the community.
  • As a founder/member, I can find people by role, company, expertise, or event relevance.
  • As a founder/member, I can understand why someone is worth meeting without reading a long profile.
  • As a visitor, I can see which people are connected to upcoming events.
  • As an organizer, I can keep people context lightweight and safe.

Affordances

  • Search input.
  • Filter chips.
  • Compact public-safe rows/cards.
  • Topic tags.
  • Related event links.
  • Optional View More / row expansion.

Person Row Content

  • Name.
  • Company / role / affiliation.
  • Location if useful.
  • Short relevance blurb.
  • Topic tags.
  • Related upcoming event(s), if any.
  • Public-safe link/contact path if allowed.

Page Design

  • Top bar matches Events and includes Companies.
  • No title banner: page title, description, and lightweight stats should move into the right sidebar on desktop.
  • Search/filter area can sit in the right sidebar on desktop and above the list on mobile.
  • Directory list uses compact rows, not social cards.
  • Row hierarchy: name → role/company → relevance/tags.
  • Right sidebar: spec-aligned rail with filters, visible fields, stats, and visibility notes.

Right Sidebar

  • Filters: topic/expertise, company, role/category, event relevance, location.
  • Visible fields: name, role/company, location, relevance blurb, tags, related events, public-safe contact path.
  • Boundaries: public-safe only, concise relevance over bios, no private-member exposure.
  • Stats: public people, topic count, event-linked people.

Filters + Non-goals

  • Filters: topic/expertise, company, role/category, event relevance, location.
  • No feed grammar: no posts, likes, follows, activity stream.
  • No messaging.
  • No LinkedIn-style exhaustive profiles.
  • No private-member exposure.
  • No self-serve account assumption.

4. Person Detail

Optional profile-lite page or row expansion when a person row needs more context.

OptionalProfile-liteNo social

User Stories

  • As a visitor, I can understand what a person works on and why they are relevant.
  • As a founder/member, I can see which events this person is connected to.
  • As a founder/member, I can decide whether this person is worth meeting or asking about a topic.
  • As an organizer, I can expose concise public context without creating a profile system.

Affordances

  • Expand/collapse row or profile-lite link.
  • Related event links.
  • Topic tags.
  • Optional external link.
  • Back to People.

Content

  • Name.
  • Role/company/affiliation.
  • Location if useful.
  • Short bio/relevance.
  • Topic tags.
  • Related events.
  • Public-safe contact/link if allowed.

Page Design

  • No title banner: person name, short bio, and key facts should move into the right sidebar on desktop.
  • If row expansion: open inline under the person row and keep it short.
  • If dedicated page: main content stays utility-first while the sidebar carries profile summary and facts.
  • Utility-first order: name → role/company → why relevant → events/topics.

Do Not Include

  • No wall/feed.
  • No private messages.
  • No follower graph.
  • No account dashboard.
  • No long biography unless explicitly useful.

5. Companies

YC Companies-style public directory for companies connected to Freddy Founders.

DirectoryYC-inspiredPublic-safe

User Stories

  • As a visitor, I can scan what each company does from a short tagline.
  • As a founder/member, I can filter companies by industry, location, stage/status, hiring/open-to-help, event relevance, or related people.
  • As a founder/member, I can understand why a company appears without reading marketing copy.
  • As a visitor, I can see which companies connect to upcoming events and public people.
  • As an organizer, I can keep company context lightweight and consent-safe.

Affordances

  • Search input.
  • YC-like filters: top/featured, hiring/open-to-help, industry, location, size/stage/status, event relevance.
  • Compact company rows/cards.
  • Logo/initial mark.
  • Tagline and metadata chips.
  • Related event links.
  • Related public founder/person links.
  • Optional View Company / row expansion.

Company Row Content

  • Logo/initial mark if available.
  • Company name.
  • Short tagline.
  • Industry/category.
  • Location/HQ or remote status.
  • Stage/status and hiring/open-to-help state.
  • Cohort/batch/community marker if used.
  • Team size range if public-safe.
  • Related upcoming event(s), if any.
  • Related public people/founders, if approved.
  • Public-safe website/contact path if allowed.

Page Design

  • Top bar matches Events and People, with Companies in public nav.
  • No title banner: page title, description, and lightweight stats move into the right sidebar.
  • Search/filter area can sit in the right sidebar on desktop and above the list on mobile.
  • Company list uses compact YC-like rows/cards, not marketing cards.
  • Row hierarchy: company name → tagline → industry/location/status → related people/events.
  • Right sidebar: context, YC-like filters, visible fields, stats, and public-safe boundaries.

Right Sidebar

  • Page context: Freddy Founders company directory, inspired by YC Startup Directory.
  • Filters: top/featured, hiring/open-to-help, industry/category, location, size/stage/status, event relevance, related people.
  • Visible fields: logo/initial, name, tagline, category, location, status, team size if public-safe, related events/people.
  • Stats: public companies, event-linked companies, hiring/open-to-help companies.
  • Boundaries: not a vendor marketplace, no private CRM notes, no implied endorsement.

Do Not Include

  • No sales/vendor marketplace.
  • No exhaustive company directory.
  • No marketing landing pages.
  • No reviews/ratings.
  • No private CRM notes.
  • No implied endorsement.

Affordances

  • Company website/contact link.
  • Related event links.
  • Related founder/person links.
  • Topic/category tags.

Do Not Include

7. Login

Simple returning-member/admin authentication entry. Public browsing remains open.

AuthUtilityNo wall

User Stories

  • As a returning member, I can log in to manage RSVP/profile context.
  • As an organizer/admin, I can log in to reach maintenance tools.
  • As a visitor, I can recover my account or move to registration.
  • As a public browser, I can understand Events, People, and Companies remain viewable without logging in.

Affordances

  • Email field.
  • Password or magic-link field.
  • Login action.
  • Forgot password / send magic link.
  • Register link.
  • Public browsing reassurance.
  • Role-aware destination after login.

Content

  • Email.
  • Password or magic-link option.
  • Login button.
  • Register link.
  • Forgot password link.
  • Privacy/public-directory reminder.

Page Design

  • No marketing hero.
  • Compact utility form.
  • Side/facts rail explains what login enables: RSVP history, profile/company edits, admin maintenance if permitted.
  • Public browsing should not feel gated.
  • Error state should not leak account existence.

Do Not Include

  • No social login wall.
  • No member feed/dashboard.
  • No complex account settings.
  • No forced login for public browsing.

8. Registration

Lightweight account creation / request-access surface with consent-aware directory context.

AuthRequest AccessConsent

User Stories

  • As a visitor, I can create or request an account without emailing the organizer.
  • As a prospective member, I can provide enough context for approval and directory safety.
  • As a founder/member, I can indicate my company affiliation and interests.
  • As an organizer, I can capture consent and avoid accidental public exposure.

Affordances

  • Name, email, and account setup fields.
  • Company/role/affiliation field.
  • Reason for joining / what building field.
  • Topic interests.
  • Public directory consent checkbox.
  • Submit registration / request access.
  • Login link.
  • Pending approval or email verification state.

Content

  • Name.
  • Email.
  • Password or account setup method.
  • Company / role / affiliation.
  • Location if useful.
  • Founder/operator context.
  • Topics/interests.
  • Consent to public profile/company display.

Page Design

  • No growth funnel.
  • Simple form, not a marketing conversion page.
  • Progressive disclosure: only ask fields needed for access, consent, and useful directory context.
  • Trust copy says public directory display is optional/approved, not automatic.
  • Post-submit state: check email or pending organizer review.

Do Not Include

  • No open social-network signup.
  • No public profile auto-publish.
  • No invite/referral gamification.
  • No onboarding wizard.

9. Admin Maintenance

Simple internal CRUD for Events, People, Companies, registrations, and relevance links.

InternalCRUDSimple

User Stories

  • As an organizer, I can create and edit an event.
  • As an organizer, I can create and edit a public-safe person row.
  • As an organizer, I can create and edit a public-safe company row.
  • As an organizer, I can link people and companies to events with relevance labels.
  • As an organizer, I can control whether people, companies, accounts, and event links are public.
  • As an organizer, I can review pending registrations and consent before approval.
  • As an organizer, I can spot missing required fields before publishing.

Affordances

  • Events table/list.
  • People table/list.
  • Companies table/list.
  • Registrations/accounts table/list.
  • Create/edit forms.
  • Link person or company to event action.
  • Approve/decline registration action.
  • Visibility/consent selector.
  • Missing-field warnings.
  • Public preview.
  • Publish/save draft actions.

Admin Objects

  • Event: title, date/time, location, host, fit/context, why attend, audience, status, RSVP, visibility.
  • Person: name, company/role, location, relevance blurb, topic tags, public-safe contact, visibility.
  • Company: name, industry/category, location, relevance blurb, topic tags, related people, public-safe website/contact, visibility.
  • Account/registration: name, email, company/role, reason for joining, approval status, consent, role.
  • Event-person / event-company link: event, person or company, relevance label, display note, visibility/consent.

Page Design

  • Internal tabs or columns for Events, People, Companies, Links, and Accounts.
  • Default view: current upcoming events, recently edited people/companies, and pending registrations.
  • Plain low-burden forms with required fields clearly marked.
  • Inline warnings are practical and specific.
  • Preview shows public event row, people row, company row, and event context links.
  • No dashboard bloat.

Do Not Include

  • No moderation system.
  • No complex issue tracker.
  • No analytics dashboard.
  • No email campaign management.
  • No payment/ticketing admin.
